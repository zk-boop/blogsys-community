# BlogSys 简历作品化与部署完善设计

## 1. 目标

将现有 Spring Boot + Vue 多用户博客完善为可长期公开访问的简历展示项目。目标运行环境为阿里云中国内地 ECS：2 vCPU、2 GiB 内存、40 GiB ESSD、3 Mbps 固定带宽。系统允许访客自行注册、投稿、评论、点赞、收藏和上传图片，同时通过限流、权限控制、七天数据保留和自动备份控制安全风险与资源消耗。

项目定位为“面向多用户内容社区的博客发布与审核系统”，不以“仿 B 站”作为产品定位。前端可以借鉴成熟内容社区的信息架构，但保留 BlogSys 自有视觉体系和产品表达。

## 2. 范围

本轮包含：

- 生产 Docker Compose、Nginx、Spring Boot、MySQL 配置；
- 密钥隔离、生产 Session Cookie、内部网络和容器资源限制；
- 数据库初始化、独立应用账户和生产结构校验；
- 注册、登录、投稿、评论和上传接口限流；
- 上传文件类型、内容、大小和路径校验；
- 演示基础数据保护与访客数据七天自动清理；
- MySQL 自动备份及恢复流程；
- 关键身份、所有权、审核状态、限流和清理测试；
- 默认封面压缩、懒加载、gzip 和静态资源缓存；
- 项目说明、部署、备份恢复和验收文档。

本轮不包含：

- Redis 或分布式限流；
- Kubernetes；
- 对象存储；
- 完整 CI/CD 平台；
- 全量 Testcontainers 测试体系；
- Java、Spring Boot、Vue 或构建工具的大版本升级。

## 3. 部署架构

```text
Internet
   |
   | 80 / 443
   v
Nginx + Vue SPA
   |
   | /api, Docker internal network
   v
Spring Boot 2.7 / Java 8
   |
   | Docker internal network
   v
MySQL 8
```

部署遵循以下约束：

- 只有 Nginx 向公网发布端口；Spring Boot 和 MySQL 不映射公网端口。
- Vue 生产环境使用同源相对地址 `/api`，不包含固定公网 IP。
- Nginx 反向代理 `/api` 到后端，并转发受信任的主机、来源协议和客户端地址信息。
- 上传文件使用独立 Docker Volume，重新构建容器不会删除文件。
- MySQL 数据使用独立 Docker Volume。
- Spring Boot 使用独立 `blogsys` 数据库用户，不使用 MySQL root 账户。
- 所有密码、允许来源、Cookie 安全参数和部署域名通过 `.env` 注入。
- JVM 最大堆约 384 MiB；MySQL 内存预算约 512 MiB；为操作系统、Docker、Nginx 和文件缓存保留资源。
- 容器配置健康检查、日志轮转和 `unless-stopped` 重启策略。
- 域名备案完成前使用公网 IP 验证；域名接入后启用 HTTPS 并将 HTTP 跳转到 HTTPS。

## 4. 数据库与配置

- 旧的 `testxwf.sql` 不再作为生产整库初始化入口。
- 结构初始化与演示数据初始化分离；生产启动不会重复导入演示数据。
- 生产环境将 Hibernate `ddl-auto` 设置为 `validate`，禁止运行时自动修改表结构。
- 本轮整理当前结构为可重复执行的基线初始化脚本，并提供从空数据库启动的验证步骤。
- 对后续结构变更预留 Flyway 接入点，但本轮不迁移为完整 Flyway 历史。
- 本地、测试和生产配置分离；测试配置不引用开发者本地数据库密码。
- `.env.example` 仅包含变量名和不可直接用于生产的占位值。

## 5. 身份与权限

系统继续使用 Spring Security、JDBC Session 和 CSRF，不引入 JWT。

- 匿名用户只能读取已发布文章及其公开信息。
- 登录用户以服务端认证上下文作为身份来源；写操作不信任客户端传入的用户 ID、作者 ID 或角色。
- 文章作者可以读取自己的草稿和待审核文章；其他普通用户不能读取。
- 普通用户只能修改和删除自己的资料、文章及评论。
- 审核、发布、退回、推荐、置顶和用户管理仅管理员可操作。
- 被禁用账户立即失去受保护接口访问能力。
- 生产 Cookie 启用 `HttpOnly`、`Secure` 和 `SameSite=Lax`。
- 管理员凭据不出现在 README、前端页面、演示脚本或日志中。

## 6. 限流设计

部署为单实例，因此采用进程内令牌桶，不引入 Redis。

| 操作 | 限制维度 | 限额 |
|---|---|---:|
| 注册 | 客户端 IP | 每小时 5 次 |
| 登录 | 客户端 IP | 每 10 分钟 20 次 |
| 投稿/提交审核 | 用户 | 每天 10 篇 |
| 评论 | 用户 | 每小时 30 条 |
| 文件上传 | 用户 | 每小时 20 次 |

- 达到限制返回 HTTP 429 和统一 API 错误结构。
- 前端针对 429 显示明确的稍后重试提示。
- 客户端 IP 只从受信任的 Nginx 代理链获取，避免任意伪造转发头绕过限制。
- 限流状态允许在容器重启后清空；这是单机简历演示站可接受的取舍。

## 7. 文件上传安全

- 只允许 JPEG、PNG、WebP。
- 单文件最大 5 MiB。
- 同时检查扩展名、声明 MIME 类型和文件内容签名；仅修改扩展名不能绕过校验。
- 文件名由服务端生成，禁止使用客户端路径片段。
- 规范化后的目标路径必须位于配置的上传根目录内。
- 上传和删除均要求登录；删除需要校验资源所有者或管理员权限。
- 清理访客数据时同步删除失去引用的头像和封面文件。

## 8. 演示数据生命周期

数据分为永久演示数据和临时访客数据。

- 管理员、预置用户、预置文章、预置分类和预置标签标记为永久数据。
- 公开注册产生的账户标记为临时访客数据。
- 临时账户创建的文章、评论、点赞、收藏和上传文件随账户归入临时数据范围。
- 每天凌晨按 Asia/Shanghai 时区运行清理任务。
- 注册时间超过七天的临时账户及其关联数据整体删除。
- 清理命令在事务中执行；文件删除在数据库清理成功后按待删除清单执行。
- 清理日志只记录开始时间、结束时间、各类删除数量和结果，不记录用户名、邮箱、正文或 Session 信息。
- `cleanup.enabled=false` 可在本地开发、问题排查或临时活动期间关闭清理。
- 永久数据不根据 ID 范围判断，而使用明确的数据保留标记，避免新增演示数据被误删。

## 9. 备份与恢复

- ECS 主机每天执行一次 `mysqldump` 压缩备份。
- 在清理任务运行前完成当日备份。
- 本机保留最近七份数据库备份，超期自动删除。
- 上传 Volume 单独按日打包，至少保留最近三份；数据库和文件备份采用相同日期标识。
- 备份目录不位于公开 Web 根目录。
- 恢复流程要求先停止写入、恢复数据库、恢复对应日期上传文件，再执行健康检查。
- 文档提供备份、列出备份、恢复和验证命令，但不包含真实密码。

## 10. 自动化测试

后端必须覆盖：

- 匿名访问受保护写接口返回 401；
- 普通用户不能修改其他用户资料、文章和评论；
- 非作者不能读取未发布文章；
- 管理员可以审核，普通用户调用审核接口返回 403；
- 草稿、待审核、已发布的允许和禁止状态转换；
- 被禁用用户不能继续写入；
- 重复点赞和收藏不产生重复记录；
- 注册、登录、投稿、评论和上传达到阈值后返回 429；
- 非法类型、伪造扩展名和超限文件被拒绝；
- 七天内临时账户保留，超过七天临时账户清理；
- 永久数据始终保留；
- 清理关联关系后不存在孤立点赞、收藏、评论或文件记录。

快速测试使用 H2；MySQL 方言相关的基线初始化、约束和清理 SQL 使用独立 Docker MySQL 验证。测试数据在每个测试内创建，不依赖固定主键或共享开发数据库。

## 11. 前端与性能

- 将约 2 MiB 的默认封面转换为 WebP，目标大小不超过约 200 KiB。
- 列表封面启用浏览器原生懒加载并提供明确尺寸，减少布局偏移。
- Nginx 对带内容哈希的 JS、CSS 和图片配置长期缓存。
- `index.html` 不做长期缓存，保证新版本入口及时更新。
- API 响应不被 Nginx 静态缓存。
- 启用 gzip，覆盖 HTML、CSS、JavaScript、JSON 和 SVG。
- Axios 使用 `/api` 并统一处理 401、403 和 429；Session 失效时清理本地状态并引导重新登录。
- 保留 Vue Router History 模式的 `try_files` 回退。
- 3 Mbps 带宽下重点控制首屏图片和静态资源体积。

## 12. 错误处理与可观察性

- API 使用一致的成功与错误响应结构。
- 参数错误返回 400，未登录返回 401，无权限返回 403，不存在或不可见资源返回 404，状态冲突返回 409，限流返回 429。
- 全局异常处理不返回堆栈、SQL、文件系统路径或原始异常消息。
- 应用日志不记录密码、Cookie、Session ID、完整邮箱或请求正文。
- 提供无需身份验证、只反映应用和数据库基本可用性的健康检查。
- Docker 日志配置大小和文件数量上限，避免占满 40 GiB 系统盘。

## 15. 验收标准

满足以下条件才可部署为公开简历站：

1. 后端完整测试命令退出码为 0，关键安全角色矩阵测试全部通过。
2. Vue 生产构建退出码为 0，生产包中不包含固定 API IP。
3. 空 MySQL 8 数据库可以完成结构和演示数据初始化。
4. Docker Compose 启动后所有容器达到健康状态，只有 Nginx 发布公网端口。
5. 注册、登录、投稿、提交审核、管理员审核、公开阅读和评论流程可完成。
6. 达到限流阈值时返回 429，未达到阈值时业务不受影响。
7. 测试时钟下能够证明七天清理边界和永久数据保护。
8. 数据库备份可以恢复，上传文件在容器重建后仍存在。
9. 代码、镜像、日志和文档中不包含真实生产密码、Cookie 或 Session ID。
10. 3 Mbps 网络条件下，默认封面不再造成 MiB 级单图下载。
11. GitHub 仓库公开可访问，默认分支为 `main`，且敏感信息扫描无命中。

## 16. 实施顺序与回滚

实施分为四个可独立验收的切片：

1. **仓库基线**：敏感信息和大文件检查、统一 `.gitignore`、移除嵌套 Git 元数据、初始化顶层仓库、MIT License 和 GitHub 远程。远程推送前保留两个子仓库的本地备份；需要回滚时恢复各自 `.git` 元数据。
2. **发布基础**：配置隔离、数据库初始化、Docker 网络、资源限制、健康检查和 Nginx。回滚时保留现有数据库与上传 Volume，恢复上一版镜像和 Compose 文件。
3. **安全与数据**：限流、上传校验、永久/临时标记、七天清理和备份。数据库变更先备份；回滚应用前保留新增字段，不执行破坏性降级。
4. **质量与展示**：权限和状态测试、资源压缩、统一前端错误处理、README、部署及备份文档。该切片不改变核心数据库结构，可单独回滚静态资源与文档。

每个切片均按测试先行、最小实现、完整回归的顺序执行；前一个切片未通过验收时不进入下一个切片。
